# `~/.glue/config.yaml` — User Configuration

Primary user-edited YAML config loaded by `GlueConfig.load()`.

## Path Resolution

Glue reads:

1. `$GLUE_HOME/config.yaml`, when `GLUE_HOME` is set
2. `~/.glue/config.yaml`, otherwise

Use `glue --where` to see the resolved path. Use `glue config init` to create
an annotated starter file, or `glue config init --force` to replace an existing
file with a fresh template.

Credentials do not need to live in `config.yaml`. Prefer environment variables
or `~/.glue/credentials.json`.

## Precedence

Precedence is field-specific, but the common order is:

1. CLI flags, where available
2. Environment variables
3. `config.yaml`
4. Built-in/catalog defaults

Examples:

- active model: `--model` → `GLUE_MODEL` → `active_model` → catalog default
- shell executable: `GLUE_SHELL` → `shell.executable` → `SHELL` → `sh`
- title generation: `GLUE_TITLE_GENERATION_ENABLED` →
  `title_generation_enabled` → `true`

This is the same shape generated by `glue config init`, with optional entries
commented out.

## Compatibility Fallbacks

The browser config parser still accepts these flat compatibility keys:

- `web.browser.anchor_api_key`
- `web.browser.hyperbrowser_api_key`

Prefer the nested form for new config:

```yaml
web:
  browser:
    anchor:
      api_key: your-anchor-key
    hyperbrowser:
      api_key: your-hyperbrowser-key
```

## Removed v1 Fields

The legacy v1 shape is rejected at startup. Do not use:

- top-level `provider`
- top-level `model`
- top-level `title_model`
- `<provider>.api_key` blocks such as `anthropic.api_key`
- profile objects shaped as `{ provider, model }`

Use `active_model: provider/model`, `small_model: provider/model`, flat
profile refs, and environment variables or `~/.glue/credentials.json` for
credentials.

For exact parsing logic, see `cli/lib/src/config/glue_config.dart`.
